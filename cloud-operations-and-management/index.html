Cloud Operations and Management

DURATION: January 2026 (12 WEEKS) | TUTOR: Dr Stelios Sotiriadis

Learning Objectives

  • To identify and learn the fundamental concepts behind Cloud Computing.
  • To identify and learn the different Cloud solutions.
  • To identify, configure and implement various level of Cloud services and tools.
  • Critically determine appropriate methodologies, tools and techniques to develop and implement resilient cloud solutions.
  • Articulate and identify the relevant standards and procedures used in implementing cloud computing services and infrastructure.

Unit 1: Introduction to Cloud Computing

Week 1 provided a comprehensive introduction to cloud computing and helped formalise my existing professional experience within academic frameworks. Although I have spent many years working with Microsoft Azure, M365 and private cloud environments, the unit seminar and formative activities allowed me to transition from a purely technical perspective to a strategic and reflective one. A key outcome was the identification of cloud computing as a specific service delivery model rather than just a different location for hosting virtual machines.

While preparing for the Week 1 seminar, I explored the differences among cloud computing models such as Infrastructure as a Service (IaaS), Platform as a Service (PaaS), and Software as a Service (SaaS). The post Navigating the Cloud Service Model Landscape discusses the distinct characteristics of these models, which reflect how they dictate the level of control and responsibility an organisation retains. My analysis highlighted that while I am highly comfortable with the granular control of IaaS and private clouds, the modern industry trend toward PaaS and SaaS offers superior scalability and lower operational overhead, provided that the associated risks are managed.

While working on the comparison between AWS and Google Cloud for the formative discussion, I completed a second activity. The post AWS vs Google Cloud: Comparison and Contrast discusses provider flexibility, pricing, and workload suitability, reflecting how different platforms are optimised for specific organisational needs. This allowed me to step outside my Microsoft ecosystem and evaluate how AWS prioritises a vast service catalogue while Google Cloud focuses on high performance and data analytics. This comparison reinforced the importance of choosing a provider based on specific workload needs rather than technical familiarity.

Reflecting on my learning goals, I have identified a need to move beyond traditional cloud management and embrace cloud native principles. While my background in Microsoft Azure, M365 and private clouds is valuable, the academic deep dive into the NIST characteristics of cloud computing, specifically rapid elasticity and measured service, has shown me where private infrastructure often falls short. My goal for this module is to leverage this foundation to develop more resilient, multi-cloud strategies that mitigate vendor lock-in and enhance business agility.

Unit 3: Cloud Design Tools

This unit covered Cloud Architecture Frameworks, the Software Development Life Cycle (SDLC), and Infrastructure as Code (IaC). The lecturecast introduced the TOGAF framework and its Architecture Development Method (ADM), which provides a structured approach for aligning IT architecture with business objectives (The Open Group, 2022). I also explored the role of IaC tools like Terraform, Ansible, and CloudFormation in automating infrastructure provisioning and reducing configuration drift.

For the formative discussion, I evaluated Terraform as a cloud design tool. The post Terraform: A Superior Cloud Design Tool argues that Terraform's multi-cloud support, declarative model, and state management make it a practical choice for MSP environments where clients operate across different providers. Feitosa et al. (2024) supported this analysis by showing that developers working with Terraform actively consider cost in their IaC configurations. In my peer response, I discussed how Terraform and Ansible complement each other, with Terraform handling provisioning and Ansible managing configuration, reflecting a layered approach described by Morris (2020).

Unit 4: Cloud Native Technology Part 1

This unit focused on scripting languages, GUI-based cloud management tools, and OpenStack as tools for automating cloud environments. The lecturecast built on the architectural frameworks from earlier weeks by introducing practical skills for managing cloud infrastructure through code.

For the formative activity, I submitted a Bash script that I use in production to automate deployment of a multi-container application on Azure. The post Automating Cloud Deployment with Bash presents the script and explains how it uses Azure CLI and Azure Container Registry Tasks to build images in the cloud and selectively deploy to Azure Container Apps based on git change detection. This activity reinforced how scripting reduces human error, saves time, and creates a traceable deployment process that can be version-controlled and reviewed by a team (Microsoft, 2024).

References

Microsoft (2024) Azure Container Apps overview. Available at: https://learn.microsoft.com/en-us/azure/container-apps/overview

Unit 5: Cloud Native Technology Part 2

This unit covered cloud-native technologies, the 12-Factor App methodology, and Kubernetes as a container orchestration platform. The lecturecast explored how cloud-native development enables scalable and flexible applications, building on the scripting and automation concepts from Unit 4.

For the formative activity, I set up a local Kubernetes cluster using Minikube and deployed a containerised application to it. The post Setting Up a Local Kubernetes Cluster with Minikube walks through the setup process and explains how working with Kubernetes directly helped me understand the orchestration layer that managed services like Azure Container Apps abstract away. Having used managed container platforms professionally without interacting with Kubernetes at this level, the exercise gave me a clearer picture of pods, deployments, and services as the building blocks of container orchestration (Kubernetes, 2024).

References

Kubernetes (2024) Minikube: Local Kubernetes. Available at: https://minikube.sigs.k8s.io/docs/

Unit 6: Hybrid Cloud Solutions (Hybrid, Fog, Edge Computing)

This unit examined hybrid cloud solutions and the challenges of integrating Fog and Edge Computing into cloud systems, along with issues like vendor lock-in when implementing hybrid strategies.

Development Team Project: Cloud Infrastructure and Framework Report

This team-based project required our group of four to evaluate a cloud infrastructure framework. We chose Microsoft Azure Resource Manager (ARM) and produced a report covering its core features, service model support across IaaS, PaaS, SaaS, and FaaS, and its application across public, private (Azure Stack Hub), and hybrid (Azure Arc) deployment types. The report included a case study of a small retail organisation migrating on-premises infrastructure to Azure, demonstrating improvements in scalability, governance, and operational resilience (Microsoft, 2024). I contributed the infrastructure design and IaC deployment sections, drawing on my professional experience with ARM templates and Bicep in client environments. Tutor feedback noted strong conceptual understanding and clear structure, while highlighting that stronger evaluation of design choices would improve analytical depth.

Formative Activity: Hybrid Cloud Architecture Design

For the formative activity, I designed a hybrid cloud architecture combining OpenStack for on-premise infrastructure with Fog computing principles for an IoT application. The design addressed how edge devices can process time-sensitive data locally before forwarding aggregated results to the cloud, reducing latency and bandwidth use. This exercise connected directly to the vendor lock-in discussion from the lecturecast, as OpenStack offers an open-source alternative that avoids dependency on a single cloud provider (Rosado and Bernardino, 2022).
