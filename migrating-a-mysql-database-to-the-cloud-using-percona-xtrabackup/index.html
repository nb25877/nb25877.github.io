Migrating a MySQL Database to the Cloud Using Percona XtraBackup

Introduction

Database migration is one of the most critical and risk-prone steps in any cloud migration project. This post walks through the process of migrating a MySQL database from a local machine to a cloud environment using Percona XtraBackup, an open-source backup tool designed for MySQL and MariaDB databases. The focus is on maintaining data consistency and minimising downtime during the migration.

Why Percona XtraBackup?

Percona XtraBackup performs hot backups of MySQL databases, meaning it can create a consistent backup without locking the database or interrupting read and write operations. This is important for production databases where downtime needs to be kept to a minimum. Unlike mysqldump, which creates a logical backup by exporting SQL statements, XtraBackup copies the actual data files at the physical level, making it faster for large databases (Percona, 2024).

Migration Steps

Step 1: Install Percona XtraBackup

Percona XtraBackup was installed on the source machine running MySQL 8.0 on Ubuntu:

sudo apt-get install percona-xtrabackup-80

Step 2: Create a Full Backup

A full backup of the local MySQL database was created:

xtrabackup --backup --target-dir=/data/backups/full \
  --user=root --password=<password>

This copies the InnoDB data files and transaction logs to the target directory without locking the database.

Step 3: Prepare the Backup

Before restoring, the backup needs to be prepared. This step applies any pending transactions captured in the log files to make the data files consistent:

xtrabackup --prepare --target-dir=/data/backups/full

Step 4: Transfer Backup to Cloud Instance

The prepared backup was transferred to the cloud-hosted MySQL instance using rsync over SSH:

rsync -avz /data/backups/full/ user@cloud-instance:/var/lib/mysql/

Step 5: Restore on the Cloud Instance

On the cloud instance, the MySQL data directory was replaced with the transferred backup files:

sudo systemctl stop mysql
sudo rsync -avz /data/backups/full/ /var/lib/mysql/
sudo chown -R mysql:mysql /var/lib/mysql
sudo systemctl start mysql

Step 6: Verify Data Consistency

After restoration, data consistency was verified by comparing row counts and checksums between the source and target databases:

# On source
mysql -e "SELECT COUNT(*) FROM app.orders;"

# On target
mysql -e "SELECT COUNT(*) FROM app.orders;"

Both returned matching results, confirming a successful migration.

Challenges Faced

Network transfer speed: For larger databases, transferring backup files over the network can be slow. Compressing the backup before transfer using gzip or pigz would reduce transfer time significantly.

Permission issues: After restoring the data directory on the cloud instance, MySQL failed to start because file ownership had changed during the transfer. This was resolved by resetting ownership to the mysql user.

Version compatibility: Percona XtraBackup must match the MySQL version on both source and target. A version mismatch between XtraBackup 8.0 and an older MySQL instance on the cloud caused an initial restore failure, which was resolved by upgrading the target instance.
