Automating Cloud Deployment with Bash: A Real-World Azure CLI Script

Introduction

This post presents a Bash script I wrote and use in production to automate the build and deployment of a multi-container application on Microsoft Azure. The script provisions cloud resources using Azure CLI commands and is part of the deployment workflow for an MSP platform that runs on Azure Container Apps with images stored in Azure Container Registry (ACR).

The full script is included below, followed by an explanation of how it works and why automation through scripting is important for cloud operations.

The Script

 
#!/usr/bin/env bash
# Fast local deploy to Azure using ACR Tasks — no local Docker needed.
# Source context is uploaded to ACR and built in the cloud.
#
# Usage:
#   ./deploy-local.sh           — auto-detect changes and deploy affected containers
#   ./deploy-local.sh --admin   — force admin only
#   ./deploy-local.sh --portal  — force portal only
#   ./deploy-local.sh --agent   — force agent only
#   ./deploy-local.sh --all     — force everything
#   ./deploy-local.sh --full    — force full rebuild including Go
#   ./deploy-local.sh --dry-run — show what would deploy without doing it

set -euo pipefail

# Config
ACR_NAME="acrnshield2"
ACR="${ACR_NAME}.azurecr.io"
IMAGE="${ACR}/mspcc"
RESOURCE_GROUP="rg-nshield-prod"
ADMIN_APP="ca-nshield-prod"
PORTAL_APP="ca-nshield-portal"
AGENT_APP="ca-nshield-agent"
SHA_FILE=".last-deployed-sha"

# Parse flags
MODE="auto"
DRY_RUN=false
FORCE_FULL=false

for arg in "$@"; do
  case $arg in
    --admin)   MODE="admin" ;;
    --portal)  MODE="portal" ;;
    --agent)   MODE="agent" ;;
    --all)     MODE="all" ;;
    --full)    FORCE_FULL=true; MODE="all" ;;
    --dry-run) DRY_RUN=true ;;
  esac
done

# Change detection using git diff
detect_changes() {
  local last_sha=""
  if [ -f "$SHA_FILE" ]; then
    last_sha=$(cat "$SHA_FILE")
  fi

  if [ -z "$last_sha" ]; then
    echo "==> No previous deploy SHA found"
    CHANGED_BACKEND=true
    CHANGED_FRONTEND=true
    CHANGED_MIGRATIONS=true
    return
  fi

  echo "==> Detecting changes since last deploy (${last_sha:0:7}..HEAD)"
  local files
  files=$(git diff --name-only "$last_sha" HEAD 2>/dev/null || true)
  local uncommitted
  uncommitted=$(git diff --name-only HEAD 2>/dev/null || true)
  local staged
  staged=$(git diff --name-only --cached 2>/dev/null || true)
  files=$(printf '%s\n%s\n%s' "$files" "$uncommitted" "$staged" | sort -u | grep -v '^$' || true)

  while IFS= read -r file; do
    case "$file" in
      src/db/migrations/*|src/db/seeds/*)  CHANGED_MIGRATIONS=true; CHANGED_BACKEND=true ;;
      src/*)                                CHANGED_BACKEND=true ;;
      apps/web/*)                           CHANGED_FRONTEND=true ;;
      agents/rmm/*)                         CHANGED_AGENT=true ;;
      package.json|package-lock.json)       CHANGED_DEPS=true ;;
      Dockerfile)                           CHANGED_DOCKER=true ;;
      *)                                    CHANGED_BACKEND=true ;;
    esac
  done <<< "$files"
}

# Build in ACR and deploy to Azure Container Apps
TAG="local-$(git rev-parse --short HEAD)-$(date +%s)"

az acr build \
  --registry "${ACR_NAME}" \
  --image "mspcc:${TAG}" \
  --image "mspcc:latest" \
  .

# Deploy to each container app as needed
az containerapp update \
  --name "${ADMIN_APP}" \
  --resource-group "${RESOURCE_GROUP}" \
  --image "${IMAGE}:${TAG}"

git rev-parse HEAD > "$SHA_FILE"
echo "Done. Image: ${IMAGE}:${TAG}"

Note: The script above is a simplified version. The full production script includes additional logic for selective deployment, agent builds, and dry-run mode.

How the Script Works

The script automates the full build and deployment cycle for a cloud application running on Azure Container Apps. It starts by checking which parts of the codebase have changed since the last deployment. It does this by comparing the current Git HEAD against a stored SHA from the previous deploy. Based on the files that changed, it maps those changes to the correct containers. For example, changes to backend source files trigger an admin container rebuild, while frontend changes trigger both admin and portal container updates.

Once the targets are identified, the script builds a Docker image using Azure Container Registry (ACR) Tasks. This means the build happens in the cloud rather than on the local machine, which saves time and avoids the need for Docker to be installed locally. The built image is tagged with the Git commit hash and a timestamp for traceability.

After the build completes, the script uses Azure CLI to update the relevant Azure Container Apps with the new image. It handles three separate container apps: admin, portal, and agent. Each one is only updated if its corresponding source files changed, which avoids unnecessary deployments.

Why Scripting Matters for Cloud Operations

Automation through scripting is essential for modern cloud operations for several reasons.

First, it removes the risk of human error. Manual deployments require running multiple commands in the right order, which is easy to get wrong, especially under pressure. A script runs the same steps every time, exactly as defined.

Second, scripting saves time. What would take 15 to 20 minutes of manual work is reduced to a single command that completes in a few minutes. This is especially important when deploying frequently during development cycles.

Third, scripts provide a record of what happened. By storing the deployed SHA and tagging images with commit hashes, the script creates a clear audit trail. If something goes wrong, it is straightforward to identify which code was deployed and roll back.

Fourth, selective deployment based on change detection reduces cloud costs. Instead of rebuilding and redeploying everything on each change, the script only targets the containers that are actually affected. This reduces build time, network usage, and container restart overhead.

Finally, scripting supports collaboration. The deployment process is defined in code, stored in version control, and can be reviewed by the team. This is far more reliable than relying on documentation or tribal knowledge about how to deploy.

In summary, this script demonstrates how Bash and Azure CLI can be combined to automate cloud resource provisioning and deployment in a way that is repeatable, efficient, and traceable.
