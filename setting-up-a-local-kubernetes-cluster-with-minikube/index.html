Setting Up a Local Kubernetes Cluster with Minikube

Introduction

As someone who works with Azure Container Apps daily, I have been using managed container orchestration without needing to think much about the underlying Kubernetes layer. This activity gave me the opportunity to set up a local Kubernetes cluster using Minikube and get hands-on with the core components that platforms like Azure abstract away.

What is Minikube?

Minikube is a lightweight tool that runs a single-node Kubernetes cluster on your local machine. It is designed for learning and development purposes, allowing you to experiment with Kubernetes without needing a cloud subscription or a multi-node setup. It supports multiple container runtimes including Docker and containerd, and runs inside a virtual machine or container on your workstation.

Setting Up the Cluster

I set this up on my Dell XPS 17 running Windows with Docker Desktop already installed. The steps were straightforward.

First, I installed Minikube using the Windows installer:

# Download and install Minikube
winget install Kubernetes.minikube

Then I started the cluster using Docker as the driver:

# Start Minikube with Docker driver
minikube start --driver=docker

# Verify the cluster is running
minikube status

# Check the node is ready
kubectl get nodes

The output confirmed a single node running with the roles of control-plane and master. From here I deployed a simple Nginx web server to test the cluster:

# Create a deployment
kubectl create deployment nginx-test --image=nginx

# Expose it as a service
kubectl expose deployment nginx-test --type=NodePort --port=80

# Get the URL to access it
minikube service nginx-test --url

This gave me a local URL where I could access the Nginx welcome page in my browser, confirming the full pipeline was working: cluster running, pod scheduled, service exposed.

I also explored the Minikube dashboard, which provides a web-based UI to view deployments, pods, services, and logs:

minikube dashboard

Setting up Minikube helped me understand what is happening beneath the managed services I use in production. When I deploy containers to Azure Container Apps, Azure handles the Kubernetes orchestration behind the scenes. Working with Minikube directly exposed the building blocks: nodes, pods, deployments, services, and the kubectl command-line interface that ties them together.

The key concepts I took away from this exercise are worth noting. Pods are the smallest deployable unit in Kubernetes, wrapping one or more containers. Deployments manage the desired state of pods, handling scaling and updates automatically. Services provide stable networking endpoints so that other applications or users can reach the running pods. These are the same primitives that power production Kubernetes clusters at scale, just running locally on a single node.

One thing that stood out was how quickly Minikube lets you iterate. I could create a deployment, break it on purpose, watch Kubernetes restart the failed pod, and see self-healing in action. This is something I rely on in production with Azure but had never seen at the Kubernetes level directly.

From an MSP perspective, understanding these fundamentals matters. Not all clients use managed services. Some run their own Kubernetes clusters on-premises or in hybrid setups, and being able to speak to the core concepts makes a difference when designing solutions or troubleshooting issues.

Conclusion

Minikube is a practical tool for learning Kubernetes without the cost or complexity of a cloud-hosted cluster. This exercise bridged the gap between the managed container services I use professionally and the orchestration layer that powers them. Going forward, I plan to use Minikube as a local testing environment before pushing container configurations to Azure.
