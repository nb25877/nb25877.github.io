Designing a Disaster Recovery Plan Using Azure Site Recovery

1. Introduction

Disaster Recovery (DR) is about making sure that when something goes wrong, whether it is a hardware failure, a regional outage, or a ransomware attack, services can be restored quickly with minimal data loss. This post walks through the process of designing and implementing a DR plan for a cloud-hosted business application using Microsoft Azure Site Recovery (ASR).

While the module's focus has been on OpenStack-based environments, I chose to work with Azure for this exercise because it is the platform I use professionally to manage client infrastructure. The principles of DR planning, including defining recovery objectives, automating failover, and testing recovery procedures, apply regardless of the cloud platform (Carson and Rodrick, 2024).

2. Scenario

The scenario involves a small professional services firm running its core line-of-business application on Azure. The environment includes:

  • Two virtual machines (application server and database server) hosted in Azure UAE North
  • Azure SQL Database for structured data
  • Azure Blob Storage for document storage
  • Users accessing the application through a web frontend

The business requires the application to be recoverable within 1 hour (RTO) with no more than 15 minutes of data loss (RPO).

3. DR Strategy Design

Recovery Objectives

ObjectiveTarget
Recovery Time Objective (RTO)Less than 1 hour
Recovery Point Objective (RPO)Less than 15 minutes
DR RegionAzure West Europe (secondary)

Components

Azure Site Recovery (ASR): Handles continuous replication of virtual machines from the primary region (UAE North) to the secondary region (West Europe). ASR replicates VM disks asynchronously, capturing changes every few minutes to meet the RPO target.

Azure SQL Geo-Replication: The Azure SQL Database is configured with active geo-replication to the secondary region. This provides an automatically updated readable secondary that can be promoted to primary during a failover.

Azure Blob Storage GRS: Blob storage is configured with Geo-Redundant Storage (GRS), which replicates data to the paired region. For faster failover, Read-Access GRS (RA-GRS) allows read access to the secondary copy before a full failover is triggered.

Azure Traffic Manager: A DNS-based traffic routing service configured with priority routing. Under normal conditions, all traffic goes to the primary region. If the health probe detects an outage, traffic is automatically redirected to the secondary region.

4. Implementation Process

Step 1: Enable ASR Replication

Azure Site Recovery was enabled in the Recovery Services vault. The two virtual machines were added to the replication policy with a replication frequency of 5 minutes. ASR automatically provisions placeholder resources (NICs, disks, availability sets) in the target region.

Step 2: Configure SQL Geo-Replication

Active geo-replication was configured on the Azure SQL Database through the Azure portal. The secondary database was created in West Europe with asynchronous replication. A failover group was set up to allow automatic failover if the primary becomes unreachable.

Step 3: Configure Storage Redundancy

The Blob Storage account was set to RA-GRS to ensure data is replicated to the paired region and remains readable during an outage.

Step 4: Set Up Traffic Manager

An Azure Traffic Manager profile was created with priority routing. The primary endpoint pointed to the UAE North frontend, and the secondary endpoint pointed to the West Europe resources. Health probes were configured to check application availability every 30 seconds.

Step 5: Create a Recovery Plan

A recovery plan was built in ASR that defines the order of operations during failover. The database server starts first, followed by the application server, ensuring dependencies are respected. Custom scripts were added to update connection strings and verify service health after failover.

5. Testing the DR Plan

A test failover was executed using ASR's built-in test failover feature. This creates the recovered VMs in an isolated virtual network in the secondary region without affecting production. The test confirmed:

  • Both VMs started successfully in the correct order
  • The application connected to the geo-replicated SQL database
  • Users could access the application through the Traffic Manager endpoint
  • Recovery completed in approximately 40 minutes, within the 1-hour RTO target

6. Reflection

What worked well

  • ASR's continuous replication and automated failover made the process straightforward compared to manual backup and restore approaches
  • The recovery plan feature ensured that VMs started in the right dependency order
  • Test failover allowed validation without impacting production

Challenges

  • Initial setup required careful configuration of networking in the secondary region to match the primary
  • SQL geo-replication added latency to write operations, which needed to be communicated to the application team
  • Cost of maintaining replicated resources in a secondary region is an ongoing expense that needs to be justified against the business impact of downtime

Comparison with open-source approaches

Tools like Duplicity and Bacula, which my peers explored using OpenStack, offer cost-effective DR for environments where managed services are not available. However, they require more manual effort during recovery and lack the automated failover capabilities that ASR provides. The trade-off is between cost and recovery speed, and the right choice depends on the organisation's budget and RTO requirements.

7. Conclusion

This exercise demonstrated that a well-designed DR plan using Azure Site Recovery can meet tight recovery objectives with relatively low operational effort. The key takeaway is that DR planning is not just about having backups. It requires automated replication, defined recovery procedures, and regular testing to be effective (Carson and Rodrick, 2024).
