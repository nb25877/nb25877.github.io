Deploying a Serverless Function with OpenFaaS

Introduction

Serverless computing allows developers to run code without managing the underlying infrastructure. Cloud providers handle provisioning, scaling, and maintenance, while developers focus on writing functions that execute in response to events. This post walks through deploying a simple serverless function using OpenFaaS, an open-source serverless platform that can run on top of Docker and Kubernetes.

Why OpenFaaS?

While managed serverless platforms like AWS Lambda and Azure Functions are widely used in production, OpenFaaS offers a provider-agnostic alternative that can run on any infrastructure where Docker or Kubernetes is available. This makes it a good option for learning how serverless works at a lower level, and for organisations that want to avoid vendor lock-in with their serverless workloads (OpenFaaS, 2024).

Setup

OpenFaaS was deployed locally using Docker Swarm. The faas-cli command-line tool was installed to manage functions.

 
bash
# Install faas-cli
curl -sL https://cli.openfaas.com | sudo sh

# Deploy OpenFaaS on Docker Swarm
docker swarm init
git clone https://github.com/openfaas/faas.git
cd faas
./deploy_stack.sh

# Log in to the OpenFaaS gateway
faas-cli login --password <password> --gateway http://127.0.0.1:8080

Creating the Function

I created a simple cloud cost estimator function that takes a number of virtual machines and returns an estimated monthly cost. This is a task I deal with regularly when scoping cloud solutions for clients.

bash
# Create a new function using the Python template
faas-cli new cost-estimator --lang python3

# This creates the following structure:
# cost-estimator/
#   handler.py
#   requirements.txt
# cost-estimator.yml

The function logic in handler.py:

import json

def handle(req):
    try:
        data = json.loads(req)
        vm_count = int(data.get("vm_count", 1))
        vm_size = data.get("vm_size", "standard")

        pricing = {
            "small": 25.00,
            "standard": 75.00,
            "large": 150.00
        }

        unit_cost = pricing.get(vm_size, 75.00)
        monthly_cost = vm_count * unit_cost

        return json.dumps({
            "vm_count": vm_count,
            "vm_size": vm_size,
            "unit_cost_usd": unit_cost,
            "estimated_monthly_cost_usd": monthly_cost
        })
    except Exception as e:
        return json.dumps({"error": str(e)})

Building and Deploying

 
bash
# Build the function container
faas-cli build -f cost-estimator.yml

# Deploy to OpenFaaS
faas-cli deploy -f cost-estimator.yml --gateway http://127.0.0.1:8080

Testing the Function

bash
# Invoke the function
echo '{"vm_count": 5, "vm_size": "standard"}' | faas-cli invoke cost-estimator

# Response:
# {"vm_count": 5, "vm_size": "standard", "unit_cost_usd": 75.0, "estimated_monthly_cost_usd": 375.0}

The function executed successfully, returning the cost estimate within milliseconds. OpenFaaS handled the container lifecycle automatically, spinning up the function on invocation and scaling it down when idle.

How Serverless Optimises Cloud Operations

Serverless architecture optimises cloud operations in several ways. First, it eliminates the need to provision and maintain servers. The platform handles scaling automatically based on demand, so resources are only consumed when the function is actually running. Second, the pay-per-invocation pricing model of managed serverless platforms means organisations only pay for the compute time they use, which can significantly reduce costs for workloads with variable traffic. Third, serverless functions are small and focused, making them easier to develop, test, and deploy independently compared to monolithic applications (Toosi et al., 2025).

The trade-off is that serverless introduces cold start latency when a function has not been invoked recently, and there are limits on execution duration and memory that may not suit all workloads. For long-running or compute-heavy tasks, traditional container-based deployments remain more appropriate.
