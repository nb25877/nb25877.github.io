Security Audit of an OpenStack Cloud Application Using OpenVAS

1. Introduction

This post presents the findings from a mock security audit of a cloud application deployed on OpenStack. The audit uses OpenVAS (Open Vulnerability Assessment Scanner) to identify vulnerabilities across the cloud infrastructure. Each finding is mapped to relevant ISO/IEC 27001 controls, and mitigation strategies are recommended.

OpenStack provides a flexible open-source platform for private and hybrid cloud deployments, but this flexibility also means that security responsibility falls heavily on the operator. Regular vulnerability assessments help ensure that misconfigurations and known weaknesses are identified before they can be exploited (OpenStack Foundation, 2023).

2. Audit Scope and Methodology

Target systems: OpenStack controller node, two compute nodes hosting a multi-tier web application, and associated networking and storage services.

Scanning approach: OpenVAS was configured to run both authenticated and unauthenticated scans across the internal management network and the tenant application network. Authenticated scans used SSH credentials on the compute nodes to assess package-level vulnerabilities. Scan results were classified using CVSS v3 severity scores and mapped to ISO/IEC 27001:2022 controls.

3. Vulnerabilities Identified

3.1 Unrestricted API Access to Keystone Endpoints

Severity: High

The Keystone identity service API was accessible from the tenant network without IP restrictions. OpenVAS identified the admin API endpoint responding to unauthenticated requests from any network segment.

Risk: An attacker with access to the tenant network could attempt token-based attacks or enumerate user accounts through the exposed admin API.

Mitigation: Restrict Keystone admin API access to the management network only using Neutron security groups and firewall rules. Separate the admin and public API endpoints onto different network interfaces.

ISO/IEC 27001 Control: A.8.20 (Network security)

3.2 Missing Disk Encryption on Compute Nodes

Severity: High

OpenVAS flagged that the block storage volumes attached to compute instances were not using encryption. Data written to Cinder volumes was stored in plaintext on the underlying storage backend.

Risk: If physical storage media is compromised or decommissioned without proper wiping, sensitive application data could be recovered.

Mitigation: Enable Cinder volume encryption using LUKS with keys managed through Barbican (OpenStack Key Manager). Apply encryption as the default for all new volumes through Cinder volume type policies.

ISO/IEC 27001 Control: A.8.24 (Use of cryptography)

3.3 Stale User Accounts in Keystone

Severity: Medium

The authenticated scan identified several Keystone user accounts that had not been used in over 90 days but retained active project roles, including two accounts with admin-level privileges.

Risk: Dormant accounts with elevated privileges are a common target for credential-based attacks. If compromised, they provide an attacker with legitimate access that is difficult to detect.

Mitigation: Implement an account lifecycle policy that disables accounts after 60 days of inactivity. Conduct quarterly access reviews to remove unnecessary role assignments. Where possible, integrate Keystone with a centralised identity provider to manage account provisioning and deprovisioning.

ISO/IEC 27001 Control: A.5.18 (Access rights)

3.4 Outdated OpenStack Services

Severity: Medium

Several OpenStack services on the controller node were running versions two releases behind the current stable branch. OpenVAS matched these against known CVEs including vulnerabilities in the Nova API and Neutron L3 agent.

Risk: Known vulnerabilities in outdated services can be exploited using publicly available tools and scripts.

Mitigation: Establish a patch management schedule aligned with OpenStack release cycles. Test updates in a staging environment before applying to production. Subscribe to the OpenStack Security Advisories (OSSA) mailing list for early notification of critical patches.

ISO/IEC 27001 Control: A.8.8 (Management of technical vulnerabilities)

3.5 No Centralised Log Aggregation

Severity: Medium

OpenStack services were writing logs locally on each node. There was no centralised log collection or correlation, and no alerting for security-relevant events such as repeated authentication failures or security group modifications.

Risk: Without centralised logging, detecting and investigating security incidents becomes slow and unreliable. Attackers can also tamper with local log files to cover their tracks.

Mitigation: Deploy a centralised logging solution such as ELK (Elasticsearch, Logstash, Kibana) or Graylog. Forward all OpenStack service logs and system audit logs to the central store. Configure alert rules for high-priority events including failed Keystone authentications and changes to security group rules.

ISO/IEC 27001 Control: A.8.15 (Logging)

5. Compliance Recommendations

To improve alignment with ISO/IEC 27001:

  1. Establish a formal vulnerability management process with scheduled OpenVAS scans.
  2. Implement encryption for data at rest across all storage services.
  3. Enforce account lifecycle management with periodic access reviews.
  4. Maintain a patching schedule tied to OpenStack security advisories.
  5. Deploy centralised logging with automated alerting for security events.

6. Conclusion

This audit identified five vulnerabilities ranging from high to medium severity. The findings highlight common security gaps in OpenStack deployments, particularly around API exposure, encryption, identity management, patching, and logging. Addressing these issues through the recommended mitigations would significantly strengthen the security posture and move the environment closer to compliance with ISO/IEC 27001.
